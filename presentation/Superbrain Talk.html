How small language models actually get made.

01
DATA QUALITY, NOT SIZE
Phi-3 was trained "data-optimal" — heavily filtered web + synthetic textbook data. Microsoft even dropped trivia like sports scores to spend capacity on reasoning.
↗ Phi-3 Technical Report · arXiv:2404.14219, 2024
02
TOKENS ≫ PARAMETERS
Llama 3.1 8B saw ~15 trillion tokens. The model is small; the training set is enormous. You push data, not parameter count.
↗ Llama 3.1 model card · Meta, 2024
03
POST-TRAINING IS THE BEHAVIOUR
Instruction-following comes from SFT + DPO — preference data, reward models, rejection sampling — layered onto the base model.
↗ Phi-3 & Llama 3.1 post-training, 2024
04
QUANTIZE TO FIT THE LAPTOP
4-bit Q4_K_M GGUF shrinks 8B weights to ~5 GB with little quality loss. That's what Ollama serves by default.
↗ Ollama model library, 2024–25
The myth, corrected Distillation, pruning, and LoRA exist — but phi3:mini and llama3.1:8b weren't made that way. The real levers were better data and post-training.

LESSON 05 · Retrieval quality is answer quality.

☁ Cloud API — what you get free
GPT-4 reasons over weak context and still produces plausible answers. Bad retrieval is masked by fluency. You need eval harness scores — not user complaints — to notice.

# failure is quiet
# you need eval to catch it
⚡ SLM · Superbrain — what we built

+ Hybrid retrieval: vector + BM25
+ RRF fusion across both
+ Per-chunk embeddings, not per-article
+ Postgres FTS index on chunk content
· Prompt tweaks — barely moved the needle
· Larger generation model — didn't either
The budget rule Spend 80% of your QA improvement budget on retrieval. The LLM can only cite what you gave it. Garbage in, hallucination out.

Auto-scales transparently. Thousands of concurrent requests distributed across infrastructure. You never think about queuing or request priority. It's someone else's problem.
/ask → 800ms
128 background jobs → all fine
# concurrency: someone else's problem
↗ "Local LLMs Handle Concurrency" — mljourney.com, 2026
↗ "Ollama: Laptop to Cluster" — bentoml.com, 2025
⚡ SLM · Superbrain — what we built
Single Ollama process. 8 reclassification jobs × 16 articles = 128 background calls. /ask blocked for 5 minutes. Grafana showed us why.
P1 · /ask QA — interactive · jumps queue
P2 · ingestion — user-triggered background
P3 · classification · digest — fully background
asyncio.PriorityQueue · single worker · FIFO within tier
The lesson A local model is a single-threaded resource. Design your queue discipline before you ship background work — not after your user waits 5 minutes for an answer.

LESSON 07 · Gate on retrieval score — before you call the model.

☁ Cloud API — the problem still exists
Cloud LLMs are fluent — plausible-sounding answers even with weak retrieval. The failure is quiet. You need eval scores, not user complaints, to catch it. The gate is available to you — you just don't feel forced to build it.
Q: "my car broke down"
top_chunk_score: 0.31
→ fluent answer. confidently wrong.
↗ "Better RAG: Similarity with Threshold" — M. Lee, Medium, 2025
⚡ SLM · Superbrain — what we built
SLMs fail loudly with weak context. That made us build the gate. The gate is right engineering for any RAG system — cloud or local.
retrieve top-k chunks via BM25 + vector
if top_score < 0.6 → abort, no LLM call
else → generate with grounded evidence
"I don't have enough on that topic.
I can talk about: AI & ML, DeepSeek R1, SLMs…"
The insight The gate is not an SLM trick. It's good RAG engineering — available to anyone. SLMs just made us build it sooner, because the failure was too loud to ignore.
